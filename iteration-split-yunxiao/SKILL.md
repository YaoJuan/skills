# 需求拆分 + 云效落库（通用）

把任意项目的产品需求，按敏捷方法拆成迭代工作项并落库到云效。封装了一套可复用的拆分方法和默认约定，项目专属信息在运行时确认，不绑定任何具体项目。

落库平台：云效 Projex，通过 `alibabacloud-devops-mcp-server`（yunxiao MCP，需在 Claude Code/Cursor 等已连接）。

## 运行前先确认项目上下文（缺则询问，不要猜）

每个项目这些可能不同，开始前先确认：

- **目标云效项目** + 是测试还是正式（首次先用测试项目）。
- **系统/产品线前缀**：多系统项目用前缀区分（如 `[A系统]`/`[B系统]`/`[共用]`）；单系统项目则省略前缀。
- **范围来源**：本期 PRD / 功能清单 / 甘特图等定稿文件。
- **本期迭代**：名称、起止日期。
- **优先级/排期**（可选）：已知的 P0/P1；缺失则标"待定"问用户。
- **合规/安全要求**（如有）：如等保三级、数据权限等，需体现在验收标准/标签。
- **工时口径**：默认 1 人天 = 8 小时（团队若不同则按团队来）。

## 默认约定（团队习惯，自动套用；项目不同可被覆盖）

1. **默认精简两层**：`模块（业务类需求）→ 职能任务`。砍掉中间需求层；同模块同职能任务合并、工时累加。**颗粒要粗**。仅当某模块特别大/复杂、用户明确要求时，才下沉到三层（模块→需求→任务）。
2. **命名**：`[系统前缀]（多系统时）职能 — 具体动作`。职能词：后端/前端/App/小程序/UI设计/联调/数据/部署/文档。
3. **优先级映射**（云效用 紧急/高/中/低）：`P0 → 高`，`P1 → 中`，`P2 → 低`。"紧急"只留给线上故障。
4. **工时**：估算用人天，登记云效按 `人天 × 8 = 小时`（云效预计工时单位是小时）。投入系数（如 0.7）只用于容量核算，不乘到任务工时。
5. **功能测试不建任务**：提测后走**看板状态流转**（→ 测试），测试时间在排期/甘特阶段层规划。只有**需要写代码的测试**（自动化用例、CI 接入、专项/性能）才建测试任务。
6. **范围以最新定稿为准**：用最新功能清单/PRD/甘特图，不用过期旧文档；已完成的功能标记为已完成、不拆分。
7. **合规要求**（若项目有）：相关需求验收须含安全/合规项并打标签。
8. **DoD（默认四环境）**：交付链路 本地→测试→UAT→生产；"做完"延伸到 UAT 验收，不止研发自测。项目环境不同可调整。

## 工作流程（带确认门）

### ① 读取与对齐
读 PRD/功能清单/甘特图，理解本期范围；确认哪些功能已完成（不拆）；确认上面的项目上下文。

### ② 生成精简两层拆分清单（先给人看，不写库）
按"模块 → 职能任务"输出清单（Markdown 表格或 HTML）。每个模块一张表，列：职能任务、人天、小时(×8)。给出系统前缀（如有）、优先级（高/中/低）、合计人天/小时。标注合规项。功能测试不列任务。
输出后**停下，请用户审核**颗粒度、优先级、工时。

### ③ 用户确认
用户调整后确认。优先级/工时/归属一律以确认后的清单为准，不自行改动。

### ④ 落库云效（确认后才执行）
类型映射：`模块 → 业务类需求`（纯技术模块如鉴权/日志/部署CI 用 `技术类需求`）；`任务 → 任务`，直接挂在模块下（无中间层）。

落库前置检查：
- `search_projects` 让用户确认目标项目。
- `get_work_item_types` 确认 业务类需求/技术类需求/任务 的 type_id（若"业务类需求"未启用，提示到 项目设置→工作项类型 开启）。
- `get_work_item` 看一条样例，确认设置"父工作项"用的字段名。

创建顺序（自顶向下，边建边关联）：
1. `create_sprint` 建迭代（名称、起止日期取自用户）。
2. 逐模块：`create_work_item` 建业务类需求(模块)，拿到 ID。
3. 在模块下 `create_work_item` 建各职能任务，**创建时即设 父工作项 = 该模块ID**，并设 所属迭代、优先级、标题前缀。
4. `create_estimated_effort` 登记预计工时（人天×8 小时）。

**关键**：父工作项与所属迭代是两回事，两个都要设；只设迭代会导致任务挂不到模块。

### ⑤ 试跑 → 批量
先只建**一个模块**作为试跑，把"模块ID → 各任务ID + 一条任务的父项/优先级/工时字段值"给用户核对父子是否真正关联。确认无误后，复述"将在【项目名】继续创建剩余 N 个模块共 M 任务"，等用户确认，再批量建其余。

## 安全红线
- 未经用户在第③步确认，不得执行第④步写库。
- 首次/不确定时先在**测试项目**跑通。
- 令牌为全模块读写，批量创建前必须复述"将在【项目名】创建 N 个工作项"待确认。
- 不删除重建已有工作项，除非用户明确要求。

## 用到的云效 MCP 工具
`search_projects` / `get_work_item_types` / `list_work_item_relation_work_item_types` / `get_work_item`（查项目与类型、父子关系、样例字段）
`create_sprint` / `list_sprints`（迭代）
`create_work_item` / `update_work_item`（建业务类需求与任务、设父项/优先级）
`create_estimated_effort`（登记预计工时，小时）
`create_workitem_attachment` / `create_work_item_comment`（挂产品稿、加说明）

## 输出清单参考格式（第②步）

每个模块一张表（示例为通用占位，按实际模块填写）：

| 职能任务 | 人天 | 小时 |
|---|---|---|
| [前缀] 后端 — 某模块接口与逻辑 | 9.5 | 76 |
| [前缀] 前端 — 某模块页面 | 5 | 40 |
| [前缀] App — 端侧适配 | 3 | 24 |

（功能测试不在此列，走状态流转；仅自动化/CI 接入等测试开发才列为任务。）

模块汇总表：模块(业务需求) / 系统(如有) / 任务数 / 人天 / 小时 / 优先级(高/中/低) / 来源对应。
