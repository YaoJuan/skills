# Codeup PR/MR 评审

本 skill 帮助评审团队成员提交到 Codeup 的合并请求。评审者是人在回路的终审者：你的任务是把这批改动整体过一遍、按统一标准给出结构化意见，供评审者据此到 Codeup 的 MR 页面写评审、通过或打回。

## 团队分支模型（评审时的前提）

- 常驻保护分支：`master`（生产）、`develop`（集成主线），均禁止直接 push。
- `feature/*`、`fix/*` 提 MR 到 `develop`；`hotfix/*` 提 MR 到 `master`。
- 因此**评审一个分支时，基线分支默认是**：`hotfix/*` → `master`，其余 → `develop`。

## 输入

- 必需：待评审的**源分支名**（如 `feature/lottery-trend-chart`）。
- 可选：基线分支。不指定时按上面的规范自动推断。
- 如果用户给的是 Codeup MR 链接而非分支名，先从页面/链接确认源分支名，再继续。

## 评审流程

### 第 1 步：准备工作区与 diff

运行 bundled 脚本，它会 fetch、把源分支签出到独立 worktree（不打断当前工作区）、并生成相对基线的 diff：

```bash
bash scripts/prepare-review.sh <源分支名> [基线分支]
```

脚本输出里会给出三个关键路径：`WORKTREE`（签出的完整代码，用于读上下文）、`DIFF_FILE`（本次 MR 的变更 diff）、`FILES_FILE`（变更文件清单）。

读取 `DIFF_FILE` 作为评审主体。**只评审 diff 中新增/修改的代码**，不要去评审未改动的既有代码。需要理解某处改动的上下文时，去 `WORKTREE` 里读对应的完整文件。

> 若环境不便跑脚本，等价做法是：`git fetch origin` 后用 `git diff origin/<base>...origin/<branch>` 取 diff（三点语法基于稳定 merge-base，等价于 Codeup 上看到的 MR 变更）。

### 第 2 步：按标准评审

把每个问题归入两档之一。**这两档的划分要和团队的自动评审卡点保持一致**，避免人工终审和机器卡点标准打架。

**阻塞性问题（应要求修改、不应合并）：**
- 逻辑错误、边界条件处理缺失
- 空指针 / 空集合 / 越界等运行期崩溃风险
- 并发问题、资源泄漏（连接、文件句柄、线程未释放等）
- 安全问题：SQL 注入、未校验的外部输入、敏感信息硬编码/泄漏、越权
- 明显的性能陷阱（如循环内查库、N+1、不必要的全量加载）
- 破坏现有功能或接口契约的改动

**建议性问题（不阻塞，仅提示）：**
- 命名、代码风格、可读性
- 可有可无的小优化、重构建议
- 注释/文档补充

判断"是否阻塞"的尺度：**只有当问题会导致 bug 到生产、安全风险、或破坏现有功能时才算阻塞**。否则一律归建议。不要因为风格洁癖把 MR 卡死——评审者团队不大、还有外包，过度严苛会让评审变成瓶颈。

### 第 3 步：输出评审报告

按下面的格式输出。结论先行，问题按文件、按严重度组织，每条要可定位（文件:行）、说清楚为什么、给出修改方向。

```
## 评审结论：✅ 可合并（仅建议）  |  ❌ 需修改后再合

源分支：feature/xxx  →  基线：develop
变更：N 个文件，+X / -Y 行

### ❌ 阻塞性问题（M 项）
1. `path/to/File.java:42` —— <问题> <为什么有问题> <怎么改>
   ...

### 💡 建议（K 项）
1. `path/to/Other.kt:18` —— <建议> <理由>
   ...

### 小结
<两三句话：整体质量、是否可合、修改后重点复查哪几处>
```

没有阻塞性问题时，结论给"可合并"，阻塞段写"无"。

### 第 4 步：清理

评审结束后移除临时 worktree：

```bash
git worktree remove --force <WORKTREE 路径>
```

## 项目相关关注点（按实际技术栈裁剪）

评审时结合本团队技术栈的常见雷区多看一眼：

- **Java / Spring Cloud 后端**：事务边界与传播、并发与线程安全、数据库连接与流的释放、MyBatis/SQL 注入、对外接口的入参校验、缓存一致性。
- **Android**：Activity/Fragment 生命周期与内存泄漏、主线程阻塞、WebView 安全配置、大图/资源加载。
- **iOS / Flutter / 移动端**：retain cycle、异步回调时的状态一致性、空安全。

这些是提示而非清单，按 diff 实际涉及的部分判断。

## 注意事项

- **基线分支不要搞错**：`hotfix` 比的是 `master`，feature/fix 比的是 `develop`。比错会让 diff 包含一堆无关改动。
- **diff 很大时**：先看 `FILES_FILE` 的文件清单分清主次，优先评审核心逻辑文件，配置/生成代码可略过；必要时分批读 diff。
- **你不直接操作 Codeup**：本 skill 只产出评审意见。是否通过、写到 MR 上、点合并，都由评审者本人在 Codeup 完成。
